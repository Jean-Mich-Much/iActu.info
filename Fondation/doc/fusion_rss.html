fusion_rss.php – Fusion et agrégation de flux RSS
Version 18 – Pipeline robuste, non bloquant

Le module fusion_rss.php assemble plusieurs flux RSS locaux en un flux unique, trié, dédupliqué et propre. Il constitue la deuxième étape du pipeline iActu V18. Le moteur est conçu pour être résilient, non bloquant et tolérant aux erreurs.

1. Rôle du module

Le module fusionne plusieurs flux RSS individuels en un flux unique, en appliquant :

  • la collecte des flux sources locaux (prefix_*.xml),
  • la lecture XML tolérante (flux cassés ignorés),
  • l’extraction des items (titre, lien, date),
  • la déduplication (URL + similarité de titres),
  • le tri par date décroissante,
  • la limitation au nombre maximal d’items,
  • la génération d’un flux RSS final propre,
  • l’écriture atomique du fichier final,
  • la journalisation légère dans un log rotatif.

Le module ne bloque jamais sur un flux ou un item invalide : il passe simplement au suivant.

2. Mode d’appel

Le module fonctionne en mode automatique unique : 

fusion_rss(string $title, string $prefix, int $min, int $max);
  • $title : titre du flux final,
  • $prefix : préfixe des fichiers sources (prefix_*.xml),
  • $min : nombre minimal d’items attendus,
  • $max : nombre maximal d’items à conserver.

Tous les fichiers correspondant à cache/rss/$prefix_*.xml sont pris en compte. Il n’existe plus de mode manuel ni de gestion de fichiers .ready.

3. Collecte des flux sources

Les flux sont recherchés dans : 

cache/rss/

Les fichiers doivent respecter le format : 

$prefix_*.xml

Le module :

  • ne crée ni ne lit de fichiers .ready,
  • n’attend jamais (aucun délai, aucun sleep),
  • ne verrouille aucun fichier (aucun lock).

Un fichier introuvable, vide ou illisible est simplement ignoré.

4. Lecture et validation minimale

Chaque fichier source est chargé via simplexml_load_file(). Un flux source est ignoré si :

  • le XML est invalide,
  • la structure <channel> / <item> est absente,
  • aucun item exploitable n’est trouvé.

Un flux cassé ne bloque jamais la fusion : le module passe simplement au suivant.

5. Extraction des items

Pour chaque flux valide :

  • parcours des balises <item>,
  • extraction des champs suivants :
  • title : texte du titre, nettoyé (trim),
  • link : URL de l’article, nettoyée (trim),
  • date : conversion via strtotime() de <pubDate> ou équivalent.

Un item est ignoré si :

  • le titre est vide,
  • le lien est vide,
  • la date est invalide (ou non convertible).

Les items valides sont ajoutés à un buffer global commun à tous les flux.

6. Déduplication – fusion_dedupe()

La déduplication repose sur deux critères complémentaires :

  • égalité stricte des URLs,
  • similarité des titres via fusion_similar_titles().

En cas de doublon, l’item le plus récent (date la plus élevée) est conservé.

7. Similarité des titres – fusion_similar_titles()

La similarité des titres repose sur les étapes suivantes :

  • normalisation via fusion_norm_title() :
    • passage en minuscules,
    • suppression de la ponctuation non alphanumérique,
    • réduction des espaces multiples,
    • trim final.
  • découpage en mots (tokens),
  • filtrage des mots de moins de 5 lettres,
  • intersection des mots significatifs.

Deux titres sont considérés comme similaires si au moins 4 mots significatifs (longueur ≥ 5 lettres) sont identiques.

Ce seuil (4 mots) offre une déduplication robuste, limitant fortement les faux positifs.

8. Tri et limitation

Une fois tous les items collectés et dédupliqués, le module applique :

  • un tri par date décroissante : 
    usort($items, fn($a, $b) => $b['date'] <=> $a['date']);
  • une limitation au nombre maximal d’items :
    • seuls les $max premiers items sont conservés,
    • le flux final doit contenir au moins $min items pour être considéré comme valide.

9. Génération du flux final – fusion_generate_rss()

Le flux final est généré sous la forme d’un RSS 2.0 propre :

  • XML en UTF‑8,
  • balises <channel> et <item> complètes,
  • titres et liens échappés via htmlspecialchars(..., ENT_XML1, 'UTF-8'),
  • dates au format RFC‑822 (DATE_RSS).

La structure est minimale mais strictement correcte pour un lecteur RSS moderne.

10. Écriture atomique

Le flux final est écrit de manière atomique afin d’éviter toute publication partielle :

  • écriture dans un fichier temporaire $prefix.xml.tmp,
  • application des permissions correctes (par exemple 664),
  • renommage atomique du fichier temporaire vers $prefix.xml.

En cas d’échec d’écriture, le fichier final n’est pas remplacé, garantissant l’absence de flux corrompu.

11. Validation finale – fusion_valid_final()

Après génération, le flux final peut être soumis à une validation minimale :

  • présence d’au moins $min items,
  • présence des balises <channel> et </rss>,
  • terminaison correcte du document.

En cas d’échec, le flux n’est pas publié (le fichier existant est conservé).

12. Gestion des logs – fusion_log()

Les opérations sont journalisées dans logs/fusion.log. Le fichier de log est géré avec une rotation légère :

  • si le fichier dépasse 512 octets, il est vidé,
  • si le fichier n’a pas été modifié depuis plus de 7 jours, il est vidé,
  • chaque nouvelle entrée est ajoutée avec un horodatage.

La journalisation reste minimale et ne bloque jamais la fusion.

13. Résumé opérationnel

  • collecte des flux locaux ($prefix_*.xml)
  • lecture XML tolérante (flux cassés ignorés)
  • extraction des items valides (titre, lien, date)
  • déduplication (URL + 4 mots significatifs ≥ 5 lettres)
  • tri par date décroissante
  • limitation à $max items
  • génération d’un flux RSS 2.0 propre
  • validation minimale du flux final
  • écriture atomique du fichier final
  • log léger avec rotation simple

14. Conclusion

Le module fusion_rss.php assure une fusion robuste, cohérente et non bloquante des flux RSS normalisés. Il garantit :

  • une agrégation fiable des flux sources,
  • une déduplication précise (URL + 4 mots significatifs),
  • une production de flux propre et stable,
  • une compatibilité totale avec PHP 8.3 / 8.4, UTF‑8 et environnements Linux.

Il constitue la deuxième étape du pipeline iActu V18, avec un design simple, résilient et adapté à une exécution régulière via cron.
