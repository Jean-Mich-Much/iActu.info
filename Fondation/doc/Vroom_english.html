🚀 Vroom: the reinvented PHP database

Vroom is a handcrafted PHP database designed to be simple, readable, fast and unbreakable. It is built on a clear principle: one record = one file, with a RAM‑centered, multitasking engine that touches the disk only through full atomic writes.

🧩 What is Vroom?

Vroom is a minimalist storage engine designed to replace an SQL database in cases where:

  • simplicity is the priority
  • read/write performance is critical
  • robustness and corruption prevention are essential
  • deployment must remain ultra‑lightweight

Vroom combines:

  • Vroom DB: physical storage in .vrec files
  • Vroom Engine: RAM buffers, transactions, locks, workers, queue, multitasking

📦 Overall architecture

📁 Vroom DB

  • One .vrec file per record
  • Solid ASCII header (id, type, timestamps, relations)
  • Data encoded in Base64
  • Automatic distribution into subfolders based on ID
  • Final marker to verify integrity

⚙️ Vroom Engine

  • RAM buffers for each opened file
  • Single‑pass atomic write
  • RAM → disk transactions
  • Simple physical locks
  • Parallel workers (pcntl_fork)
  • Queue, supervision, timeouts
  • RAM cache with LRU eviction

🧠 How does Vroom work?

1. Reading

When a file is requested:

  • disk read → RAM
  • header and data stored in buffers
  • relations automatically preloaded
  • RAM cache updated

2. Modification

All modifications happen in RAM:

  • no partial writes
  • no file kept open for extended writing
  • no corruption risk

3. Atomic write

During commit:

  • full file built in RAM
  • written to a temporary file
  • atomic rename()
  • final marker written last

4. Automatic cleanup

After each write:

  • removal of orphan .tmp files
  • removal of expired .lock files
  • removal of corrupted .vrec files

🛠️ Full Vroom API

Command Description Example
create(type,data): int Creates a record $id=create('post',['title'=>'Hello']);
get(type,id): ?array Reads a record $post=get('post',$id);
all(type): array Lists all records $posts=all('post');
findBy(type,field,value): array Simple search $p=findBy('post','user_id',1);
update(type,id,data): bool Updates a record update('post',$id,['title'=>'OK']);
delete(type,id): bool Deletes a record delete('post',$id);
begin() Starts a RAM transaction begin();
commit() Commits writes commit();
rollback() Rolls back writes rollback();
info() Full internal state var_dump(info());

⚡ Concrete examples

Create a post

<?php
$id=create('post',[
'title'=>'Hello',
'body'=>'Content',
'user_id'=>1
]);
?>

Read and modify

<?php
$p=get('post',$id);
$p['title']='Updated';
update('post',$id,$p);
?>

Transactions

<?php
begin();
$id1=create('post',['title'=>'A']);
$id2=create('post',['title'=>'B']);
commit();
?>

Workers and queue

<?php
vroom_job_enqueue(function(){
return ['ok'=>true];
});
vroom_job_run_queue(2);
?>

🧬 Detailed internal structure

This section describes in depth the role of each internal Vroom file. It is intended for developers, architects and curious readers who want to understand the exact behavior of the engine.

📌 Vroom.php

Main entry point. Provides the public API:

  • create(), get(), update(), delete()
  • all(), findBy()
  • begin(), commit(), rollback()
  • info()
  • job management (vroom_job_*)

Loads all internal modules and orchestrates the entire engine.

🧠 Vroom_state.php

The brain of the engine:

  • global state
  • RAM buffers
  • LRU cache
  • active transactions
  • queue
  • active workers
  • internal statistics

🔢 Vroom_id.php

Full ID management:

  • unique ID generation
  • atomic increment
  • subfolder structure (00/12/34/...)
  • path normalization

🔒 Vroom_lock.php

Physical lock management:

  • creation of .lock files
  • automatic expiration
  • prevention of concurrent writes
  • cleanup of orphan locks

📖 Vroom_disk_read.php

Disk → RAM reading:

  • secure file reading
  • ASCII header extraction
  • Base64 decoding
  • final marker verification
  • RAM cache update

✍️ Vroom_disk_do.php

RAM → disk writing:

  • full file assembly in RAM
  • write to .tmp
  • atomic rename()
  • final marker write
  • cleanup of orphan .tmp files
  • cleanup of corrupted .vrec files

💾 Vroom_ram.php

RAM buffer management:

  • temporary storage
  • RAM modifications
  • rollback and commit
  • LRU eviction
  • RAM ↔ disk synchronization

⚡ Vroom_exec.php

Multitasking management:

  • workers via pcntl_fork()
  • timeouts
  • supervision
  • job isolation

📨 Vroom_job.php

Queue management:

  • vroom_job_enqueue()
  • vroom_job_run_queue()
  • sequential or parallel execution
  • job logs

🔗 Vroom_relations.php

Record relationship management:

  • parent → children relations
  • reverse relations
  • automatic preloading
  • optional cascade deletion
  • link consistency

📁 Internal directories

📂 data/

Physical storage:

<type>/00/12/34/00001234.vrec
  • one file per record
  • ASCII header
  • Base64 data
  • final marker

📂 logs/

Internal logs:

  • jobs
  • errors
  • expired locks
  • detected corruptions
