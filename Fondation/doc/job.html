job.php – Gestionnaire de jobs asynchrones
Version 18 – Exécution contrôlée, logs, fork et stabilité

Ce document décrit le fonctionnement du module job.php, utilisé par iActu V18 pour exécuter les parsers RSS et les fusions de manière contrôlée, stable et non bloquante.

Le système de jobs permet :

  • d’empiler des appels de fonctions ;
  • de les exécuter en parallèle (CLI) ou en arrière‑plan (FPM) ;
  • de limiter le nombre de processus simultanés ;
  • de gérer les timeouts ;
  • de journaliser toutes les exécutions ;
  • d’assurer la stabilité globale du pipeline RSS.

1. Rôle général du module

Le module job.php fournit une API simple pour :

  • capturer des appels de fonctions (job_call()) ;
  • les exécuter plus tard (fin_job()) ;
  • gérer automatiquement les forks, timeouts et logs ;
  • adapter le comportement selon l’environnement :
    • CLI → exécution parallèle via pcntl_fork() ;
    • FPM/HTTP → exécution via job_worker.php en arrière‑plan.

Il constitue la couche d’orchestration utilisée par maj_cli.php et maj_fusion_cli.php.

2. Constantes principales

Le module définit plusieurs constantes de contrôle :

  • JOB_MAX = 20 → nombre maximum de processus simultanés en mode CLI.
  • JOB_TIMEOUT = 59 → durée maximale d’exécution d’un job avant kill forcé.
  • JOB_LOG_MAX = 512 → taille maximale du fichier de log avant rotation.
  • JOB_IS_CLI → vrai si le script tourne en ligne de commande.

3. Initialisation et logs

Le module crée automatiquement :

  • le dossier logs/ ;
  • le fichier job.log ;
  • les variables globales :
    • _JOB_QUEUE – file d’attente des jobs ;
    • _JOB_CAPTURE – mode capture ou exécution directe ;
    • _JOB_LOG – chemin du fichier log.

3.1 Fonction job_log()

La fonction job_log() :

  • écrit un message horodaté dans job.log ;
  • réinitialise le fichier si :
    • sa taille dépasse 512 octets ;
    • il date de plus d’une semaine.

4. Début et fin de capture

4.1 debut_job()

Active le mode capture :

  • vide la file d’attente ;
  • active _JOB_CAPTURE ;
  • log : DEBUT JOB.

4.2 fin_job()

Désactive la capture et lance l’exécution :

  • désactive _JOB_CAPTURE ;
  • log : FIN JOB ;
  • appelle _job_exec().

5. job_call() – Ajout d’un job

La fonction job_call($f, $a) ajoute un job à la file d’attente.

  • si capture active → le job est stocké ;
  • si capture inactive → exécution immédiate (rare).

Chaque job est un tableau : 

[
  'f' => 'nom_de_fonction',
  'a' => [arguments]
]

6. Exécution des jobs : _job_exec()

La fonction _job_exec() choisit automatiquement le mode d’exécution :

  • CLI + pcntl_fork() → exécution parallèle ;
  • FPM/HTTP → exécution via job_worker.php.

7. Mode FPM/HTTP : _job_exec_fpm()

En environnement web, chaque job est exécuté via : 

php job_worker.php f arg1 arg2 ... &

Caractéristiques :

  • exécution totalement asynchrone ;
  • aucun blocage de la requête HTTP ;
  • pause de 8 ms entre chaque lancement ;
  • pas de fork → compatible mutualisé / hébergement classique.

8. Mode CLI : _job_exec_cli()

En mode CLI, le module utilise pcntl_fork() pour exécuter plusieurs jobs en parallèle.

8.1 Boucle principale

  • tant qu’il reste des jobs ou des processus actifs ;
  • lance jusqu’à JOB_MAX processus simultanés ;
  • surveille les processus actifs.

8.2 Gestion des timeouts

Si un job dépasse 59 secondes :

  • kill forcé via posix_kill(SIGKILL) ;
  • log : TIMEOUT nom_fonction pid:X.

8.3 Gestion des erreurs

  • exceptions capturées ;
  • log détaillé ;
  • sortie propre du processus enfant.

9. job_worker.php

En mode FPM, les jobs sont exécutés par un script séparé :

  • charge l’environnement minimal ;
  • exécute la fonction demandée ;
  • journalise le résultat ;
  • se termine immédiatement.

Ce mécanisme garantit une exécution stable même en environnement web.

10. Mécanismes de stabilité

Le module job.php intègre plusieurs mécanismes pour garantir la stabilité :

  • file d’attente → contrôle total des jobs ;
  • fork limité → max 20 processus ;
  • timeouts → aucun job bloquant ;
  • logs automatiques → rotation + horodatage ;
  • exécution asynchrone → aucune attente côté HTTP ;
  • fallback FPM → compatible partout ;
  • séparation worker → isolation des erreurs.

11. job_worker.php – Exécuteur secondaire (mode FPM)

Le fichier job_worker.php est un module optionnel utilisé uniquement en environnement non‑CLI (FPM/HTTP). Il sert d’exécuteur isolé pour les jobs lorsque le système ne peut pas utiliser pcntl_fork().

Il est appelé automatiquement par _job_exec_fpm() via une commande du type : 

php job_worker.php fonction arg1 arg2 ... &

12.1 Rôle principal

Ce worker a trois objectifs :

  • isoler l’exécution d’un job dans un processus indépendant ;
  • éviter les blocages dans les environnements web où le fork est impossible ;
  • appliquer un timeout strict identique au mode CLI.

Il garantit ainsi une exécution stable même sur des hébergements mutualisés ou des environnements PHP limités.

12.2 Fonctionnement interne

Le worker :

  • charge les modules essentiels :
    • job.php
    • lit_rss.php
    • fusion_rss.php
  • vérifie qu’il tourne bien en CLI ;
  • récupère la fonction à exécuter et ses arguments ;
  • installe une alarme (pcntl_alarm()) pour appliquer JOB_TIMEOUT ;
  • exécute la fonction via call_user_func_array() ;
  • journalise toute erreur ou timeout ;
  • désactive l’alarme et termine immédiatement.

12.3 Timeout du worker

En cas de dépassement du délai :

  • le signal SIGALRM est déclenché ;
  • le worker écrit dans les logs : TIMEOUT WORKER nom_fonction ;
  • le processus se termine avec un code d’erreur.

12.4 Pourquoi un worker séparé ?

Le worker existe pour garantir la compatibilité maximale du système :

  • en CLI → exécution parallèle via pcntl_fork() ;
  • en FPM/HTTP → exécution isolée via job_worker.php.

Cela permet à iActu V18 de fonctionner :

  • sur serveurs dédiés ;
  • sur VPS ;
  • sur hébergements mutualisés ;
  • sur environnements restreints sans extensions POSIX.

Le worker est donc un mécanisme de fallback robuste, garantissant que tous les jobs peuvent être exécutés, quel que soit l’environnement.

12. Conclusion

Le module job.php est un orchestrateur robuste, conçu pour gérer efficacement des dizaines de jobs RSS en parallèle.

Il garantit :

  • une exécution stable ;
  • une tolérance aux erreurs ;
  • une compatibilité CLI/FPM ;
  • une gestion fine des ressources ;
  • une intégration parfaite avec le pipeline V18.

Il constitue la fondation de l’exécution asynchrone d’iActu.
